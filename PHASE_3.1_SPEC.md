# Phase 3.1 — Daily CRO Intelligence Briefing: Build Spec

**Goal.** Turn the editorial layer (sections 09–13) from always-on curated content
into a **stable daily briefing** generated on a schedule, persisted, and served
from storage — while the data layer (01–08) keeps refreshing live. Two snapshots
per day (morning + late-evening IST). Never blank; graceful staleness.

This spec matches the implemented code.

---

## 1. Two-clock model

| Layer | Sections | Refresh | Source |
|---|---|---|---|
| **Data** | 01–08 | live, per existing API cadence (Yahoo ~5m, FRED ~30m) | every page load |
| **Editorial** | 09–13 | **frozen daily snapshot** ×2 | read from storage; generated by cron |

The dashboard route **never regenerates editorial on page load** — it reads the
stored snapshot. Only the cron generates.

## 2. Snapshot schedule (Vercel Hobby = 2 crons, once/day each)

| Cron | UTC schedule | ≈ IST | Slot | Also does |
|---|---|---|---|---|
| morning | `30 0 * * *` | ~06:00 | `morning` | weekly summary on Mondays |
| evening | `0 15 * * *` | ~20:30 | `evening` | — |

- Morning runs after the full prior US close (US closes ~01:30 IST) → captures the complete previous session.
- Evening runs mid-US-session → captures Asia/Europe close + the US open reaction.
- Hobby cron timing is approximate (within the hour) and the displayed timestamp is the **actual** generation time, labelled by slot ("Morning briefing" / "Evening update").
- Weekly summary is folded into the **Monday morning** run to stay within Hobby's 2-cron limit. (On Pro, split into a third cron.)

## 3. Pipeline

```
news adapters → de-dup → cluster → relevance score (CRO topic set)
  → LLM interpretation (grounded, schema-validated) → snapshot → persist
```

- **Sources** behind adapters: Marketaux (implemented), Finnhub / NewsAPI / GNews (stubs). Premium (Reuters/FT/WSJ/Nikkei) deferred — same adapter contract.
- **Ranking** is by CRO relevance (banking / credit / market / liquidity / capital / Japan), never popularity. Earnings, single-stock, celebrity-CEO and retail noise are suppressed.
- **Gating:** the live path runs only if a news key **and** `ANTHROPIC_API_KEY` are present. Otherwise the generator emits a **curated seed snapshot** (the v3 content) so the app is fully functional from day one.

## 4. LLM reliability rules (the real work)

1. **Grounding** — the model may only interpret/cluster the stories it is given; it must not introduce facts or numbers. Each theme cites its source cluster.
2. **Schema validation** — generated JSON is validated against the `CroTheme` / `EditorialCard` shapes before persistence. Invalid/empty → discard, fall back to last good snapshot, then to seed.
3. **Derived confidence** — not guessed: `High` = data-anchored + multi-source cluster; `Medium` = single-source / emerging; `Low` = speculative.
4. **Frozen anchors** — anchor values (JGB, CPI, 10Y…) are captured **at generation time** and stored, so the briefing's prose and numbers always agree even as live data drifts during the day.

## 5. Persistence & history

- Store: **Vercel KV / Upstash (REST)**. Falls back to in-memory for local dev (not durable — KV required for the production guarantee).
- Keys: `snapshot:latest`, `snapshot:{YYYY-MM-DD}:{slot}`, `snapshot:index` (rolling), `weekly:latest`.
- Retain **14 days × 2 slots**; prune older. History powers the Weekly Summary and future theme-evolution / "New Since Yesterday".
- Every theme carries a stable `topicId` (taxonomy) so future day-over-day diffs match themes reliably instead of by fuzzy title.

## 6. Snapshot metadata (top of the editorial layer)

`EDITORIAL SNAPSHOT` block shows: **Generated** (timestamp + slot label), **Sources used**, **Articles reviewed**, **Themes generated**, **Confidence**, and a **Coverage** grid (Inflation, Central Banks, Credit, Banking, Geopolitics, Japan, China — ✓ covered / ⚠ thin). Purpose: transparency into the briefing's breadth and quality.

## 7. Failure & low-news handling

- **Generation fails** → serve the most recent successful snapshot with a banner "Using previous successful editorial snapshot" + its original timestamp. Never blank.
- **Low news volume** (weekend / holiday) → lighter briefing; do **not** manufacture themes; carry forward important themes with a "news flow was limited" flag. Monday's context compares against Friday.

## 8. Sections (unchanged structure, enhanced)

09 radar — 7 theme classes (Macro / Credit / Market / Banking / Regulatory / Strategic / Japan).
10 conversation — 3–5 ranked themes, each with Why it matters, Banking impact, **Why this matters to Mizuho**, lens box, **Signals to watch**, **Risk horizon**, and (Learning) questions / meeting prep / plain-English, plus **Confidence**.
11 editorial cards · 12 Japan & Asia watch · 13 Weekly learning summary (now generated from stored history).

## 9. Env / setup

```
# News (pick one to start)
MARKETAUX_API_KEY=
# LLM interpretation
ANTHROPIC_API_KEY=
LLM_MODEL=claude-sonnet-4-6        # optional override
# Cron protection (required)
CRON_SECRET=<random string>
# Storage (Vercel KV / Upstash) — auto-injected when you add a KV store
KV_REST_API_URL=
KV_REST_API_TOKEN=
```

Without news/LLM keys the app serves the curated seed. Without KV it works but the
daily-freeze guarantee is best-effort (in-memory). Add a KV store in Vercel →
Storage for the real guarantee.

## 10. Success criteria

Each morning, in 5–10 minutes, the user can answer: what happened, why it matters,
what CROs are discussing, what leadership may ask, what to watch next, what matters
for a Japanese global bank, which risks are immediate vs structural, and how to
contribute confidently — from a stable briefing, not a churning feed.
